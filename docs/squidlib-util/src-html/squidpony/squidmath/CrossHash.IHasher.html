010/**
011 * Simple hashing functions that we can rely on staying the same cross-platform.
012 * The static methods of this class (not its inner classes) use the Fowler/Noll/Vo
013 * Hash (FNV-1a) algorithm, which is public domain. The hashes this returns are always
014 * 0 when given null to hash. Arrays with identical elements of identical types will
015 * hash identically. Arrays with identical numerical values but different types will
016 * hash differently (only for FNV-1a, not the inner classes). FNV-1a may be somewhat
017 * slow if you are running many hashes or hashing very large data; in that case you
018 * should consider one of the inner classes, Lightning or Storm.
019 * <br>
020 * There are two static inner classes in CrossHash, Lightning and Storm, that provide different
021 * hashing properties, as well as the inner IHasher interface.
022 * <br>
023 * IHasher values are provided as static fields, and use FNV-1a to hash a specific type or fall
024 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
025 * parts of OrderedMap and OrderedSet, and allow arrays to be used as keys in those collections
026 * while keeping hashing by value instead of the normal hashing by reference for arrays. You
027 * probably won't ever need to make a class that implements IHasher yourself.
028 * <br>
029 * The inner classes provide alternate, faster hashing algorithms. Lightning has no theoretical
030 * basis or grounds in any reason other than empirical testing for why it does what it does, and
031 * this seems to be in line with many widely-used hashes (see: The Art of Hashing,
032 * http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, it performs
033 * very well, just slightly behind Arrays.hashCode (18-20 ms instead of 15 ms for over a million
034 * hashes of 16-element long arrays, including overhead for generating them, while SipHash and
035 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data).
036 * <br>
037 * Storm is a variant on Lightning with 64 bits for a salt-like modifier as a member variable,
038 * which can make 2 to the 64 individual hashing functions from one set of code. Storm has some
039 * properties of a cryptographic hash, but is not recommended it for that usage. It is, however
040 * ideal for situations that show up often in game development where end users may be able to see
041 * and possibly alter some information that you don't want changed (i.e. save data stored on a
042 * device or in the browser's LocalStorage). If you want a way to verify the data is what you
043 * think it is, you can store a hash, using one of the many-possible hash functions this can
044 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
045 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
046 * then it is unlikely they can make the hash match even if they can edit it. Storm is slightly
047 * slower than Lightning, at about 25-26 ms for a million hashes (still including some RNG
048 * overhead) instead of Lightning's 18-20 ms, but should never be worse than twice as slow as
049 * Arrays.hashCode, and is still about three times faster than the similar SipHash that SquidLib
050 * previously had in this class.
051 * <br>
052 * All of the hashes used here have about the same rate of collisions on identical data
053 * (testing used Arrays.hashCode, FNV-1a, Lightning, Storm, and the now-removed SipHash), but
054 * Lightning (and Storm) seem to do better than the rest on collision rates more often than
055 * they do worse than all the others. Lightning has been changed frequently but is considered
056 * stable now, but it isn't being considered to replace the FNV-1a algorithm in CrossHash for
057 * compatibility reasons. It seems to meet all the criteria for a good hash function, though,
058 * including doing well with a visual test that shows issues in FNV-1a and especially
059 * Arrays.hashCode. Storm is still not necessarily in a final version; the commit that removed
060 * SipHash also changed Storm's algorithm to more heavily factor in the salt and allow that
061 * salt to alter any patterns present in the output, making similar data hash very differently
062 * when the salts are different.
063 * <br>
064 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
065 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
066 * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing SipHash)
067 * does approximately as well as Lightning, with no clear patterns visible (Sip has been removed from SquidLib
068 * because it needs a lot of code and is slower than Storm and especially Lightning). The idea is from a
069 * technical report on visual uses for hashing, http://www.clockandflame.com/media/Goulburn06.pdf .
070 * <ul>
071 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
072 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
073 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
074 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
075 * </ul>
076 * <br>
077 * Note: This class was formerly called StableHash, but since that refers to a specific
078 * category of hashing algorithm that this is not, and since the goal is to be cross-
079 * platform, the name was changed to CrossHash.
080 * Created by Tommy Ettinger on 1/16/2016.
081 *

086 */

089        if (data == null)
090            return 0;

092        for (int i = 0; i <= len; i++) {
093            o |= (data[i]) ? (1 << (i & 7)) : 0;
094            if ((i & 7) == 7 || i == len) {
095                h ^= o;
096                h *= 16777619;
097                o = 0;
098            }
099        }
100        return h;
101    }

599    /**
600     * An interface that can be used to move the logic for the hashCode() and equals() methods from a class' methods to
601     * an implementation of IHasher that certain collections in SquidLib can use. Primarily useful when the key type is
602     * an array, which normally doesn't work as expected in Java hash-based collections, but can if the right collection
603     * and IHasher are used.
604     */
605    public interface IHasher extends Serializable {
606        /**
607         * If data is a type that this IHasher can specifically hash, this method should use that specific hash; in
608         * other situations, it should simply delegate to calling {@link Object#hashCode()} on data. The body of an
609         * implementation of this method can be very small; for an IHasher that is meant for byte arrays, the body could

657        if (left == right)
658            return true;
659        if ((left == null) ^ (right == null))
660            return false;
661        for (int i = 0; i < left.length && i < right.length; i++) {
662            if (!inner.areEqual(left[i], right[i]))
663                return false;
664        }
665        return true;
666    }

983    /**
984     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
985     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
986     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
987     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
988     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
989     * state by another prime called the "output multiplier", then XORs the current result with that value
990     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
991     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
992     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
993     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
994     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
995     * number drawn from the highest 6 bits of the state.
996     * <br>
997     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
998     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
999     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
1000     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
1001     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
1002     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
1003     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
1004     * gets 135 collisions. Storm depends on the salt chosen, but with one initialized with the phi-based
1005     * constant that shows up in LightRNG and here, Storm gets 116 collisions. Dispersion is not perfect, but
1006     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
1007     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
1008     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
1009     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
1010     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
1011     * <br>
1012     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
1013     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
1014     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
1015     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
1016     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
1017     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
1018     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
1019     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
1020     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
1021     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
1022     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
1023     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
1024     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
1025     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
1026     * <br>
1027     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
1028     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
1029     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
1030     * <br>
1031     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
1032     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
1033     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
1034     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
1035     * from SquidLib because it needs a lot of code and is slower than Storm and especially Lightning). The
1036     * idea is from a technical report on visual uses for hashing,
