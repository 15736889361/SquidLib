005/**
006 * Simple hashing functions that we can rely on staying the same cross-platform.
007 * The static methods of this class (not its inner classes) use the Fowler/Noll/Vo
008 * Hash (FNV-1a) algorithm, which is public domain. The hashes this returns are always
009 * 0 when given null to hash. Arrays with identical elements of identical types will
010 * hash identically. Arrays with identical numerical values but different types will
011 * hash differently (only for FNV-1a, not the inner classes). FNV-1a may be somewhat
012 * slow if you are running many hashes or hashing very large data; in that case you
013 * should consider one of the inner classes, Lightning or Storm.
014 * <br>
015 * There are two static inner classes in CrossHash, Lightning and Storm, that provide different
016 * hashing properties, as well as the inner IHasher interface.
017 * <br>
018 * IHasher values are provided as static fields, and use FNV-1a to hash a specific type or fall
019 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
020 * parts of OrderedMap and OrderedSet, and allow arrays to be used as keys in those collections
021 * while keeping hashing by value instead of the normal hashing by reference for arrays. You
022 * probably won't ever need to make a class that implements IHasher yourself.
023 * <br>
024 * The inner classes provide alternate, faster hashing algorithms. Lightning has no theoretical
025 * basis or grounds in any reason other than empirical testing for why it does what it does, and
026 * this seems to be in line with many widely-used hashes (see: The Art of Hashing,
027 * http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, it performs
028 * very well, just slightly behind Arrays.hashCode (18-20 ms instead of 15 ms for over a million
029 * hashes of 16-element long arrays, including overhead for generating them, while SipHash and
030 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data).
031 * <br>
032 * Storm is a variant on Lightning with 64 bits for a salt-like modifier as a member variable,
033 * which can make 2 to the 64 individual hashing functions from one set of code. Storm has some
034 * properties of a cryptographic hash, but is not recommended it for that usage. It is, however
035 * ideal for situations that show up often in game development where end users may be able to see
036 * and possibly alter some information that you don't want changed (i.e. save data stored on a
037 * device or in the browser's LocalStorage). If you want a way to verify the data is what you
038 * think it is, you can store a hash, using one of the many-possible hash functions this can
039 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
040 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
041 * then it is unlikely they can make the hash match even if they can edit it. Storm is slightly
042 * slower than Lightning, at about 25-26 ms for a million hashes (still including some RNG
043 * overhead) instead of Lightning's 18-20 ms, but should never be worse than twice as slow as
044 * Arrays.hashCode, and is still about three times faster than the similar SipHash that SquidLib
045 * previously had in this class.
046 * <br>
047 * All of the hashes used here have about the same rate of collisions on identical data
048 * (testing used Arrays.hashCode, FNV-1a, Lightning, Storm, and the now-removed SipHash), but
049 * Lightning (and Storm) seem to do better than the rest on collision rates more often than
050 * they do worse than all the others. Lightning has been changed frequently but is considered
051 * stable now, but it isn't being considered to replace the FNV-1a algorithm in CrossHash for
052 * compatibility reasons. It seems to meet all the criteria for a good hash function, though,
053 * including doing well with a visual test that shows issues in FNV-1a and especially
054 * Arrays.hashCode. Storm is still not necessarily in a final version; the commit that removed
055 * SipHash also changed Storm's algorithm to more heavily factor in the salt and allow that
056 * salt to alter any patterns present in the output, making similar data hash very differently
057 * when the salts are different.
058 * <br>
059 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
060 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
061 * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing SipHash)
062 * does approximately as well as Lightning, with no clear patterns visible (Sip has been removed from SquidLib
063 * because it needs a lot of code and is slower than Storm and especially Lightning). The idea is from a
064 * technical report on visual uses for hashing, http://www.clockandflame.com/media/Goulburn06.pdf .
065 * <ul>
066 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
067 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
068 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
069 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
070 * </ul>
071 * <br>
072 * Note: This class was formerly called StableHash, but since that refers to a specific
073 * category of hashing algorithm that this is not, and since the goal is to be cross-
074 * platform, the name was changed to CrossHash.
075 * Created by Tommy Ettinger on 1/16/2016.
076 *
077 * @author Glenn Fowler
078 * @author Phong Vo
079 * @author Landon Curt Noll
080 * @author Tommy Ettinger
081 */

084        if (data == null)
085            return 0;
086        int h = -2128831035, len = data.length - 1, o = 0;
087        for (int i = 0; i <= len; i++) {
088            o |= (data[i]) ? (1 << (i & 7)) : 0;
089            if ((i & 7) == 7 || i == len) {
090                h ^= o;
091                h *= 16777619;
092                o = 0;
093            }
094        }
095        return h;
096    }

730    /**
731     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
732     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
733     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
734     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
735     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
736     * state by another prime called the "output multiplier", then XORs the current result with that value
737     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
738     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
739     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
740     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
741     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
742     * number drawn from the highest 6 bits of the state.
743     * <br>
744     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
745     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
746     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
747     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
748     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
749     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
750     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
751     * gets 135 collisions. Storm depends on the salt chosen, but with one initialized with the phi-based
752     * constant that shows up in LightRNG and here, Storm gets 116 collisions. Dispersion is not perfect, but
753     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
754     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
755     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
756     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
757     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
758     * <br>
759     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
760     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
761     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
762     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
763     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
764     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
765     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
766     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
767     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
768     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
769     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
770     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
771     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
772     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
773     * <br>
774     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
775     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
776     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
777     * <br>
778     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
779     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
780     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
781     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
782     * from SquidLib because it needs a lot of code and is slower than Storm and especially Lightning). The
783     * idea is from a technical report on visual uses for hashing,
784     * http://www.clockandflame.com/media/Goulburn06.pdf .
785     * <ul>
786     * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
787     * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
