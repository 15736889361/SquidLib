007/**
008 * A quasi-random number generator that goes through one of many sub-random sequences found by J.G. van der Corput.
009 * More specifically, this offers both the normal van der Corput sequence, which only changes the state by incrementing
010 * it (this works better in the normal usage as part of a 2D or 3D point generator) and a kind of scrambled van der
011 * Corput sequence, where the state changes unpredictably (this works better when using this to generate 1D sequences,
012 * or when the base may be larger than 7 or non-prime). In both cases, the state is internally stored in a 64-bit long
013 * that is incremented once per generated number, but when scramble is true, the state is altered with something similar
014 * to a Gray code before being used; more on this later if you want to read about it. The important things to know about
015 * this class are: size of state affects speed (prefer smaller seeds, but quality is sometimes a bit poor at first if
016 * you start at 0); the base (when given) should be prime and moderately small (or very small if scramble is false, any
017 * of 2, 3, 5, or 7 should be safe); this doesn't generate very random numbers when scramble is false (which can be good
018 * for making points that should not overlap), but it will seem much more random when scramble is true; and this is a
019 * StatefulRandomness with an additional method for generating quasi-random doubles, {@link #nextDouble()}.
020 * <br>
021 * This generator allows a base (also called a radix) that changes the sequence significantly; a base should be prime,
022 * and this performs a little better in terms of time used with larger primes, though quality is also improved by
023 * preferring primes that aren't very large relative to the quantity of numbers expected to be generated. Unfortunately,
024 * performance is not especially similar to conventional PRNGs; smaller (positive) state values are processed more
025 * quickly than larger ones, or most negative states. At least one 64-bit integer modulus and one 64-bit integer
026 * division are required for any number to be produced, and the amount of both of these relatively-non-cheap operations
027 * increases linearly with the number of digits (in the specified base, which is usually not 10) needed to represent
028 * the current state. Since performance is not optimal, and the results are rather strange anyway relative to PRNGs,
029 * this should not be used as a direct substitute for a typical RandomnessSource (however, it is similar to, but simpler
030 * than, {@link squidpony.squidmath.SobolQRNG}). So what's it good for?
031 * <br>
032 * A VanDerCorputSequence can be a nice building block for more complicated quasi-randomness, especially for points in
033 * 2D or 3D, when scramble is false. There's a simple way that should almost always "just work" as the static method
034 * {@link #halton(int, int, int)} here. If it doesn't meet your needs, there's a little more complexity involved. Using
035 * a VanDerCorputQRNG with base 3 for the x-axis and another VanDerCorputQRNG with base 5 for the y-axis, requesting a
036 * double from each to make points between (0.0, 0.0) and (1.0, 1.0), has an interesting trait that can be desirable for
037 * many kinds of positioning in 2D: once a point has been generated, an identical point will never be generated until
038 * floating-point precision runs out, but more than that, nearby points will almost never be generated for many
039 * generations, and most points will stay at a comfortable distance from each other if the bases are different and both
040 * prime (as well as, more technically, if they share no common denominators). This is also known as a Halton sequence,
041 * which is a group of sequences of points instead of simply numbers. The choices of 3 and 5 are just examples; any two
042 * different primes will technically work for 2D, but patterns can become noticeable with primes larger than about 7,
043 * with 11 and 13 sometimes acceptable. Three VanDerCorputQRNG sequences could be used for a Halton sequence of 3D
044 * points, using three different prime bases, four for 4D, etc. SobolQRNG can be used for the same purpose, but the
045 * points it generates are typically more closely-aligned to a specific pattern, the pattern is symmetrical between all
046 * four quadrants of the square between (0.0, 0.0) and (1.0, 1.0) in 2D, and it probably extends into higher dimensions.
047 * Using one of the possible Halton sequences gives some more flexibility in the kinds of random-like points produced.
048 * <br>
049 * Because just using the state in a simple incremental fashion puts some difficult requirements on the choice of base
050 * and seed, we can use a technique like Gray codes to scramble the state. The sequence of these Gray-like codes for the
051 * integers from 0 to 16 is {@code 0, 9, 21, 16, 50, 51, 23, 10, 4, 28, 49, 39, 246, 198, 179, 97, 393}, and this can be
052 * generated efficiently with {@code (i * i) ^ ((i * 137) >>> 4)}. No duplicate results were found in any numbers of 20
053 * bits or less, and trying to find duplicates in 24 bits exhausted the testing computer's memory, without finding a
054 * duplicate. You should be fine.
055 * <br>
056 * Expected output for {@link #nextDouble()} called 33 times on an instance made with {@code new VanDerCorputQRNG(11, 83L, true)}:
057 * 0.5194317328051362, 0.8590943241581859, 0.5931288846390274, 0.5045420394781778, 0.7892220476743392,
058 * 0.6645037907246772, 0.7809575848644218, 0.0725360289597705, 0.6322655556314459, 0.6998838877125879,
059 * 0.5578853903421898, 0.5969537599890717, 0.6323338569769824, 0.873505908066389, 0.09514377433235435,
060 * 0.6623864490130456, 0.4376750221979373, 0.6946929854518133, 0.4250392732736835, 0.5294720305990028,
061 * 0.4790656375930606, 0.2626869749334062, 0.056075404685472306, 0.9758213236800765, 0.676729731575712,
062 * 0.899118912642579, 0.1876237961887849, 0.652755959292398, 0.039683081756710606, 0.10504746943514787,
063 * 0.2598183184208729, 0.4078273341984837, 0.3034628782187009
064 * <br>
065 * Expected output for {@link #nextDouble()} called 33 times on an instance made with {@code new VanDerCorputQRNG(19, 83L, true)}:
066 * 0.8944452544102639, 0.7842327790609341, 0.4352023081468067, 0.0696971324652205, 0.5957213342439054,
067 * 0.8864342661581787, 0.0167739658228528, 0.8956192785506557, 0.26943470353972115, 0.3525371966145134,
068 * 0.09754375733765087, 0.8924118139056637, 0.3585147443619984, 0.9126771587081131, 0.28926266679967155,
069 * 0.8542138258607592, 0.8789987799356972, 0.7905019145034184, 0.1624220194749887, 0.9349836173755572,
070 * 0.48623015477167914, 0.6716799287912155, 0.8240344994283346, 0.6557883994137552, 0.4561966221867542,
071 * 0.07946532024769608, 0.15134168706501638, 0.1382202407900492, 0.7890439760284221, 0.7142517322611092,
072 * 0.3069037223471275, 0.7030256060036372, 0.01678163918324752
073 * <br>
074 * Note on Gray-like code implementation: This is not a typical Gray code. Normally, the operation looks like
075 * {@code i ^ (i >>> 1)}, which gives negative results for negative values of i (not wanted here), and also clusters
076 * two very similar numbers together for every pair of sequential numbers. An earlier version scrambled with a basic
077 * Gray code, but the current style, {@code (i * i) ^ ((i * 137) >>> 4)}, produces much more "wild and crazy" results,
078 * but never negative ones. The period of this, if it is seen as a RandomnessSource, is probably about 2^56, at least
079 * for reasonably small starting states (under 1000 or so?). This is ideal for a scramble.
080 * <br>
081 * Created by Tommy Ettinger on 11/18/2016.
082 */
083public class VanDerCorputQRNG implements StatefulRandomness, RandomnessSource, Serializable {
084    private static final long serialVersionUID = 5;
085    public long state;
086    public final int base;
087    public final boolean scramble;
088    /**
089     * Constructs a new van der Corput sequence generator with base 13, starting point 83, and scrambling enabled.
090     */
091    public VanDerCorputQRNG()
092    {
093        base = 13;
094        state = 83;
095        scramble = true;
096    }
097
098    /**
099     * Constructs a new van der Corput sequence generator with the given starting point in the sequence as a seed.
100     * Usually seed should be at least 20 with this constructor, but not drastically larger; 2000 is probably too much.
101     * This will use a base 13 van der Corput sequence and have scrambling enabled.
102     * @param seed the seed as a long that will be used as the starting point in the sequence; ideally positive but low
103     */
104    public VanDerCorputQRNG(long seed)
105    {
106        base = 13;
107        state = seed;
108        scramble = true;
109    }
110
111    /**
112     * Constructs a new van der Corput sequence generator with the given base (a.k.a. radix; if given a base less than
113     * 2, this will use base 2 instead) and starting point in the sequence as a seed. Good choices for base are between
114     * 10 and 60 or so, and should usually be prime. Good choices for seed are larger than base but not by very much,
115     * and should generally be positive at construction time.
116     * @param base the base or radix used for this VanDerCorputQRNG; for most uses this should be prime but small-ish
117     * @param seed the seed as a long that will be used as the starting point in the sequence; ideally positive but low
118     * @param scramble if true, will produce more-random values that are better for 1D; if false, better for 2D or 3D
119     */
120    public VanDerCorputQRNG(int base, long seed, boolean scramble)
121    {
122        this.base = base < 2 ? 2 : base;
123        state = seed;
124        this.scramble = scramble;
125    }
126
127    /**
128     * Gets the next quasi-random long as a fraction of {@link Long#MAX_VALUE}; this can never produce a negative value.
129     * It is extremely unlikely to produce two identical values unless the state is very high or is negative; state
130     * increases by exactly 1 each time this, {@link #next(int)}, or {@link #nextDouble()} is called and can potentially
131     * wrap around to negative values after many generations.
132     * @return a quasi-random non-negative long; may return 0 rarely, probably can't return {@link Long#MAX_VALUE}
133     */
134    @Override
135    public long nextLong() {
136        // when scrambling the sequence, intentionally uses a non-standard Gray-like code
137        long s = (scramble) ? ++state : (++state * state) ^ (state * 137 >> 4),
138                num = s % base, den = base;
139        while (den <= s) {
140            num *= base;
141            num += (s % (den * base)) / den;
142            den *= base;
143        }
144        return (Long.MAX_VALUE / den) * num;
145    }
146
147    @Override
148    public int next(int bits) {
149        return (int)(nextLong()) >>> (32 - bits);
150    }
151
152    /**
153     * Gets the next quasi-random double from between 0.0 and 1.0 (normally both exclusive; only if state is negative or
154     * has wrapped around to a negative value can 0.0 ever be produced). It should be nearly impossible for this to
155     * return the same number twice unless floating-point precision has been exhausted or a very large amount of numbers
156     * have already been generated. Certain unusual bases may make this more likely, and similar numbers may be returned
157     * more frequently if scramble is true.
158     * @return a quasi-random double that will always be less than 1.0 and will be no lower than 0.0
159     */
160    public double nextDouble() {
161        // when scrambling the sequence, intentionally uses a non-standard Gray-like code
162        long s = (scramble) ? ++state : (++state * state) ^ (state * 137 >> 4),
163                num = s % base, den = base;
164        while (den <= s) {
165            num *= base;
166            num += (s % (den * base)) / den;
167            den *= base;
168        }
169        return num / ((double)den);
170    }
171
172    @Override
173    public VanDerCorputQRNG copy() {
174        return new VanDerCorputQRNG(base, state, scramble);
175    }
176
177    @Override
178    public long getState() {
179        return state;
180    }
181
182    @Override
183    public void setState(long state) {
184        this.state = state;
185    }
186
187    @Override
188    public String toString() {
189        return "VanDerCorputQRNG with base " + base +
190                ", scrambling " + (scramble ? "on" : "off") +
191                ", and state 0x" + StringKit.hex(state);
192    }
193
194    @Override
195    public boolean equals(Object o) {
196        if (this == o) return true;
197        if (o == null || getClass() != o.getClass()) return false;
198
199        VanDerCorputQRNG that = (VanDerCorputQRNG) o;
200
201        return state == that.state && base == that.base && scramble == that.scramble;
202
203    }
204
205    @Override
206    public int hashCode() {
207        int result = (int) (state ^ (state >>> 32));
208        result = 31 * result + base;
209        result = 31 * result + (scramble ? 1 : 0);
210        return result;
211    }
212
213    /**
214     * Convenience method that gets a quasi-random Coord between integer (0,0) inclusive and (width,height) exclusive.
215     * This is roughly equivalent to creating two VanDerCorputQRNG generators, one with
216     * {@code new VanDerCorputQRNG(2, index, false)} and the other with {@code new VanDerCorputQRNG(3, index, false)},
217     * then getting an x-coordinate from the first with {@code (int)(nextDouble() * width)} and similarly for y with the
218     * other generator. The advantage here is you don't actually create any objects using this static method, other than
219     * a (almost always shared) reference to the returned Coord. You might find an advantage in using values for index
220     * that start higher than 20 or so, but you can pass sequential values for index and generally get Coords that won't
221     * be near each other; this is not true for all parameters to Halton sequences, but it is true for this one.
222     * @param width the maximum exclusive bound for the x-positions of Coord values this can return
223     * @param height the maximum exclusive bound for the y-positions of Coord values this can return
224     * @param index an int that, if unique, positive, and not too large, will usually result in unique Coord values
225     * @return a Coord that usually will have a comfortable distance from Coords produced with close index values
226     */
227    public static Coord halton(int width, int height, int index)
228    {
229        int s = (index & 0x7fffffff)+1,
230                numX = s & 1, numY = s % 3, denX = 2, denY = 3;
231        while (denX <= s) {
232            numX *= 2;
233            numX += (s % (denX * 2)) / denX;
234            denX *= 2;
235        }
236        while (denY <= s) {
237            numY *= 3;
238            numY += (s % (denY * 3)) / denY;
239            denY *= 3;
240        }
241        return Coord.get(numX * width / denX, numY * height / denY);
242    }
243    /**
244     * Convenience method that gets a quasi-random Coord3D between integer (0,0,0) inclusive and (width,height,depth)
245     * exclusive. This is roughly equivalent to creating three VanDerCorputQRNG generators, one with
246     * {@code new VanDerCorputQRNG(2, index, false)} another with {@code new VanDerCorputQRNG(3, index, false)},
247     * and another with {@code new VanDerCorputQRNG(5, index, false)}, then getting an x-coordinate from the first with
248     * {@code (int)(nextDouble() * width)} and similarly for y and z with the other generators. The advantage here is
249     * you don't actually create any objects using this static method, other than a returned Coord3D. You might find an
250     * advantage in using values for index that start higher than 20 or so, but you can pass sequential values for index
251     * and generally get Coord3Ds that won't be near each other; this is not true for all parameters to Halton
252     * sequences, but it is true for this one.
253     * @param width the maximum exclusive bound for the x-positions of Coord3D values this can return
254     * @param height the maximum exclusive bound for the y-positions of Coord3D values this can return
255     * @param depth the maximum exclusive bound for the z-positions of Coord3D values this can return
256     * @param index an int that, if unique, positive, and not too large, will usually result in unique Coord3D values
257     * @return a Coord3D that usually will have a comfortable distance from Coord3Ds produced with close index values
258     */
259    public static Coord3D halton(int width, int height, int depth, int index)
260    {
261        int s = (index & 0x7fffffff)+1,
262                numX = s & 1, numY = s % 3, denX = 2, denY = 3, numZ = s % 5, denZ = 5;
263        while (denX <= s) {
264            numX *= 2;
265            numX += (s % (denX * 2)) / denX;
266            denX *= 2;
267        }
268        while (denY <= s) {
269            numY *= 3;
270            numY += (s % (denY * 3)) / denY;
271            denY *= 3;
272        }
273        while (denY <= s) {
274            numZ *= 5;
275            numZ += (s % (denZ * 5)) / denZ;
276            denZ *= 5;
277        }
278        return Coord3D.get(numX * width / denX, numY * height / denY, numZ * depth / denZ);
279    }
280}
